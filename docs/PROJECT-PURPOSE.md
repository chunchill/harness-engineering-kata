# Harness Engineering Kata：项目初衷与学习目标

> 本文档是本项目的**初衷文档**，用于与 AI 协同编码时提供上下文。请在做需求拆分、架构决策和 Harness 设计时优先参考本文档。

---

## 一、我们想通过这个项目达成什么

1. **深刻感受 Harness Engineering 理念**
  通过一个真实可运行的全栈示例，体会「马（AI）— 缰绳（Harness）— 骑手（人）」的分工：人设计环境与约束，AI 在约束内写代码并自检。
2. **体会 Middle Loop 过程**
  把工程师的重心放在「监督与设计环」（Middle Loop），而不是陷在逐行编码/审查（Inner Loop）或只做发布运维（Outer Loop）。重点练习：拆任务、写约束、验收 Agent 产出、从失败中改进 Harness。
3. **完整体验人—AI—DevOps 闭环**
  - **AI 编码**：在约定好的约束与上下文中，由 Agent 生成并自检代码；  
  - **DevOps 交付**：CI/CD、测试、构建、可选的部署与可观测；  
  - **人机反馈**：意图 → Agent 实现 → 自动化验证 → 人的审查与 Harness 迭代。

---

## 二、Harness Engineering 在本项目中的定位（简要）

- **Harness**：约束、反馈环、文档、Lint、生命周期管理等「让 AI 可靠工作」的基础设施。  
- **工程师角色**：设计 Harness、指定意图、提供反馈、审查结果并改进 Harness；而非亲手写每一行业务代码。  
- **三大支柱**：上下文工程（Context）、架构约束（Constraints）、熵管理（文档与代码一致性）。

---

## 三、Inner / Outer / Middle Loop 的理解（对齐 Martin Fowler）

三个 Loop 都嵌套在 **How Loop**（「如何把软件做出来」）内部。


| Loop            | 含义                                                       | 人的角色                                     |
| --------------- | -------------------------------------------------------- | ---------------------------------------- |
| **Inner Loop**  | 生成与测试代码；Agent 写代码、跑测试、自检。                                | 避免「In the loop」——不陷于逐行审代码。               |
| **Middle Loop** | 拆解任务、定义约束与验收标准、验收 Agent 产出、**改进 Harness**（而非改 artefact）。 | **On the loop**：设计并维护 Harness，校准信任、定义约束。 |
| **Outer Loop**  | 交付与发布：CI/CD、部署、运维、监控。                                    | 定发布策略、看交付与运行指标。                          |


**要点**：工程师更注重 **Middle Loop** = 更注重「On the loop」的监督与设计；当结果不对时，优先改 Harness（规则、文档、检查），再让 Agent 重跑，而不是自己改那几行代码。

---

## 四、示例项目定位：全栈 + 可体会 Middle Loop

- **项目类型**：全栈示例（如轻量任务看板 / Task Board），具备前端、后端 API、持久化与简单 UI。  
- **技术栈**：选用常见且「Harness 友好」的栈（如 React + TypeScript + Vite / Node 或 FastAPI + SQLite），便于写约束与 CI。  
- **Harness 设计要点**：  
  - 依赖分层（如 Types → Config → Repo → Service → Runtime → UI）并可在 CI 中检查；  
  - 上下文工程：`AGENTS.md`、`docs/`（架构、API 契约）作为单源真相；  
  - 验证与反馈：Pre-commit、CI（lint、test、build）、Agent 自检要求；  
  - 熵管理：API 与文档同步、架构说明与代码一致。

---

## 五、实践中如何「体会 Middle Loop」

1. **少做**：不在 Inner Loop 逐行审代码；不只在 Outer Loop 点发布。
2. **多做**：
  - **拆任务**：把需求拆成 Agent 可执行的小任务与验收标准；  
  - **写/改 Harness**：维护 `AGENTS.md`、架构与 API 文档、规则；  
  - **验收结果**：看 Agent 的 PR/输出是否「任务完成 + 符合规范」；  
  - **从失败改 Harness**：结果不对时优先改规则/文档/检查，再让 Agent 重跑。
3. **可选刻意练习**：例如「只通过改 Harness 修一个 Agent 反复犯的错」或「只通过改任务描述与验收标准让 Agent 一次做对」。

## 六、开发过程中要体会的「工作形态」与「编程范式」

### 6.1 工作形态：从 0→1、迭代完善与 Bug 修复

在本项目中，我们希望**同时**经历三种典型工作形态，以便在 Harness 与 Middle Loop 下体会人机协作的差异：

| 形态 | 含义 | 在本项目中的体现 |
|------|------|------------------|
| **从 0 到 1 的创造性开发** | Greenfield：从无到有搭架子、定架构、实现首批功能。 | 项目启动阶段：选定技术栈、定义分层与 API、实现 MVP（如看板与任务的 CRUD）。 |
| **基于现有 Feature 的完善与变更** | 在已有能力上做增强、修改、重构，而非从零写。 | 中后期：加新字段、改交互、扩展 API、调整架构边界；体会「改现有代码」时 Harness 与文档如何保持一致性。 |
| **Bug 修复** | 发现问题、定位、修复、验证，是交付过程中必然出现的环节。 | 贯穿全程：功能 bug、集成问题、回归；可刻意练习「通过改 Harness（测试、约束、复现步骤）再让 Agent 修」的 Middle Loop 方式。 |

三者会交织出现：0→1 搭好骨架后，后续以「完善/变更 + Bug 修复」为主，Harness 与 Middle Loop 的价值在演进与修 bug 时会更明显。

### 6.2 编程范式：Vibe Coding 与 Spec-Driven Development

我们预期在本次开发中会**同时触及**两种常见的人机协作范式，并对比它们在 Harness 下的表现：

| 范式 | 核心思路 | 人的投入重点 | 在本项目中的用法 |
|------|----------|--------------|------------------|
| **Vibe Coding** | 用自然语言描述想法，较少事先写死「怎么做」；人在 Why Loop，Agent 在 How Loop 里自由发挥。 | 描述目标与感受（「像 XX 那样」「别太复杂」），通过多轮对话迭代结果。 | 适合探索性功能、UI 调优、快速试错；可体会「约束少时 Agent 容易飘，需 Harness 收束」。 |
| **Spec-Driven Development (SDD)** | 先写清规格、契约或验收标准（如 API 契约、验收场景），再让 Agent 按规格实现。 | 写规格、接口定义、验收条件；Harness 中的文档与测试即「可执行规格」。 | 适合 API、领域逻辑、与已有架构对齐的改动；可体会「规格清晰 + Harness 校验」带来的可预测性。 |

- **不必二选一**：同一项目里可以部分需求用 Vibe（偏探索），部分用 SDD（偏契约）；重点是在两种方式下都**有意识地运用 Middle Loop**（拆任务、写/改约束与文档、验收结果、从失败改 Harness）。  
- **与初衷的衔接**：无论哪种范式，工程师都更关注「设计环境与约束」（Middle Loop），而不是陷在逐行写码或逐行审码（Inner Loop）。

## 七、与 AI 协同时的使用方式

- 在提出新需求或做架构/约束决策时，可引用本文档作为「项目初衷与目标」的上下文。  
- 优先在 **Middle Loop** 上协作：明确任务与验收标准、更新 Harness 与文档，再让 Agent 在 Inner Loop 中实现与自检。  
- 当 Agent 行为偏离目标时，优先考虑：是任务描述不清、约束缺失，还是文档/规则需要更新，并相应更新 `AGENTS.md` 或 `docs/` 中的内容。
